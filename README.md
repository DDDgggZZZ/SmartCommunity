一、为了满足“不少于10张表”且逻辑严密的要求，建议设计如下。请直接把这个发给后端同学去建表。
1. 基础资源类 (4张)
community_building (楼栋表)：ID, 楼号, 总层数, 描述。
community_unit (单元表)：ID, 楼栋ID(FK), 单元号 (如 1单元)。
community_room (房屋表)：ID, 单元ID(FK), 房号 (如 301), 面积, 户型, 状态(已售/未售)。
parking_space (车位表)：ID, 车位号, 状态(闲置/已租/已售)。
2. 人员与资产类 (3张)
owner_info (业主表)：ID, 姓名, 电话, 身份证号, 入住时间。
owner_room_rel (业主-房屋关联表)：ID, 业主ID, 房屋ID, 关系类型(户主/租客)。(这是典型的多对多关系，符合范式设计)
parking_usage (车位使用表)：ID, 车位ID, 业主ID, 车牌号, 到期时间。
3. 核心业务类 (3张)
fee_type (费用类型表)：ID, 名称 (物业费/水费/停车费), 单价, 单位。
fee_bill (账单表)：ID, 业主ID, 费用类型ID, 金额, 账单月份, 状态(未缴/已缴), 支付时间。
repair_request (报修工单表)：ID, 业主ID, 内容, 提交时间, 状态(待处理/处理中/完成), 维修反馈。
4. 系统管理类 (2张)
sys_user (管理员表)：ID, 用户名, 密码, 角色。
sys_notice (公告表)：ID, 标题, 内容, 发布时间。

二、 功能模块规划（要做的页面和接口）
前端需要做这些页面，后端需要写对应的 API。
1，登录/仪表盘：登录页、首页大屏（展示统计图表）。
2，房产管理：楼栋/房屋的增删改查。
3，业主管理：录入业主，绑定房子（一个业主可以有多套房）。
4，车位管理：车位录入，车位出租/出售登记。
5，收费管理：生成账单（重点功能）。缴费记录查询（把状态从“未缴”改为“已缴”）。
6，报修管理：查看报修列表，点击“处理完成”。
7，系统设置：管理员管理，公告发布。
